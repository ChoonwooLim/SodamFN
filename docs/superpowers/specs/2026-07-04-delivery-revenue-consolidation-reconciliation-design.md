# 배달앱 매출 데이터 정합화 · 4채널 엑셀 파서 · 정산 대조 — 설계

**작성일**: 2026-07-04
**상태**: 승인됨 (사장님 확인 2026-07-04)

## 배경 / 문제

매출관리 → 배달앱 화면(`/revenue`, `DeliveryAppView`)의 채널별 매출·정산·수수료가
월마다 들쭉날쭉하고 "알 수 없는 값"이 보인다. 원인 조사 결과 `DeliveryRevenue`
테이블에 **성격이 다른 레코드 3종이 채널당 중복** 적재돼 있고, 화면 병합 로직이
비결정적으로(dict iteration last-write-wins) 그중 하나를 골라 표시한다.

| 종류 | 채널명 예 | source | 성격 |
|------|----------|--------|------|
| 진짜 엑셀 | `쿠팡이츠`, `배달의민족` | `auto_coupang_excel` / `excel` | 실 매출내역서 파싱 — 총비용(중개+배달비+멤버십+쿠폰) |
| 가짜 추정 | `Coupang`, `Baemin`, `Yogiyo` (영문) | `excel` | 매출 × 고정 28% 공식 추정값 (실제 아님) |
| 입금만 | 한글명 일부 | `bank_sync` | 통장 입금액만, 매출·수수료 0 |

증상 예: 쿠팡 수수료율이 달마다 28%/58%/69% 랜덤, 요기요 매출 0인데 정산액만 표시.

**핵심 결정 (사장님)**: 수수료 = **총비용** (중개+배달비+멤버십+쿠폰+할인 등 배달앱에
나간 돈 전부). 순수 중개수수료(28%)가 아니라 실제 비용 전액.

**추가 발견**: 배민 정산명세서(.xlsx)는 **비밀번호로 암호화**돼 있어 기존 파서가
읽지 못해 배민 데이터가 불완전(1월 34건, 실제 ~350건)했던 것으로 추정.

## 목표

1. 화면 값이 **결정적·일관**되게 나오고, 4채널 모두 **총비용 기준**으로 정확.
2. 4채널(쿠팡·배민·요기요·땡겨요) 모두 **엑셀 업로드로 진짜 총비용** 적재 가능.
3. 엑셀 정산액과 통장 실입금을 **대조**해 불일치 시 이유를 사장님이 알 수 있게.

## 비목표 (이번 범위 밖)

- 요기요·땡겨요 **API 자동수집**(포털 스크래핑). 이번엔 **수동 엑셀 업로드**만.
  포털 실현가능성 조사는 후속. (플랜 B: 업로드 먼저, 자동은 나중)
- 손익계산서 계산식 자체 변경 (매출 반영 경로는 기존 `reflect_channel` 재사용).

## 파트 1 — 데이터 정리 + 결정적 병합

### 1.1 가짜 추정 레코드 제거
- `DeliveryRevenue` 중 **영문 채널명**(`Coupang`/`Baemin`/`Yogiyo`)이면서 breakdown이
  고정 28% 공식(중개11.78%+결제2.8%+배달11.2%+광고2.24%)인 레코드를 **삭제**.
- 삭제 전 건수·금액을 로그로 남기고, **전/후 비교표를 사장님께 먼저 제시**(승인 후 실행).
- 정리 스크립트: `scripts/cleanup_fake_delivery_revenue.py` (dry-run 기본, `--apply` 로 실행).

### 1.2 결정적 병합 (`/revenue/delivery-summary`)
채널 표시명 정규화(`LEGACY_CHANNEL_MAP`)는 유지하되, (year, month, 표시채널)마다
**우선순위로 단 하나의 대표 레코드**를 결정적으로 선택:

```
source_rank: auto_coupang_excel=3, excel=3, manual=2, bank_sync=0, 기타=1
대표 선택 키 = (매출>0 여부, source_rank, total_fees) 최댓값
정산액 = 그 (채널,월) 레코드들의 settlement_amount 중 non-zero 최댓값
매출   = 대표.total_sales (>0) else DailyExpense delivery 합계
수수료율 = total_fees / 매출
```
- 기존 `total_sales` override 버그(라인 412) 제거 → 대표 레코드 기준 일관.
- 함수를 `_consolidate_delivery(session, bid, year)` 로 분리(테스트 가능 단위).

### 1.3 검증
- 단위 테스트: 중복 레코드 세트 → 대표 선택이 결정적이고 총비용(엑셀) 우선인지.
- 정리 후 `/revenue/delivery-summary` 응답이 화면 표와 일치.

## 파트 2 — 4채널 엑셀 파서 정비 + 업로드 UI

### 2.1 공통
- 모든 파서는 `(year, month, 매출, 총비용, 정산액, 주문건수, fee_breakdown{})` 을
  산출해 `DeliveryRevenue(source="excel")` upsert (채널당 1행, 기존 dedup 패턴 유지).
- 매출 반영은 기존 `reflect_channel` 인라인 호출 재사용.
- 암호화 파일 지원: `msoffcrypto`(설치 확인됨)로 복호화 후 `openpyxl`.
  - 업로드 UI에 **비밀번호 입력란(선택)** 추가.
  - 자동수집용으로 채널 자격증명에 **비번을 암호화 저장**(cookie 와 동일 `encrypt_text`).
    비번 평문은 로그·문서·응답에 절대 미노출.

### 2.2 채널별 컬럼 매핑

**쿠팡** — 기존 `coupang_eats_service.sync_monthly_excel` 유지 (변경 없음).

**배민** (`baemin_service` 재구현) — 암호화 .xlsx (msoffcrypto 필요):
- `요약` 시트: (A)주문중개 (B)배달 (C)그외 (D)기타 (E)부가세 (F)우리가게클릭 (G)배민오더 (H)입금금액
  - ⚠️ 요약은 "입금완료건만" 합산 → **매출(gross)은 `상세` 시트 주문금액 합**으로 계산.
- `상세` 시트: 입금일·정산대상기간·입금금액·주문금액·중개이용료·고객할인·배달팁할인·
  배달비·결제정산수수료·조정금액·부가세·우리가게클릭·(H)입금금액·상태.
  - 매출 = Σ 주문금액, 정산 = Σ (H)입금금액, 총비용 = 매출 − 정산.
  - fee_breakdown = {중개이용료, 배달비, 배달팁할인, 결제수수료, 우리가게클릭, 부가세, 조정}.

**요기요** (신규 `yogiyo_service`) — .xlsx (비암호):
- `요약` 시트: A 주문금액(=매출), C 차감금액(=총비용), D 정산금액(=정산),
  C-1~C-14 항목(주문중개이용료·배달대행이용료·외부결제이용료·추천광고·요타임딜·쿠폰·할인 등).
- `상세 거래내역` 시트: 주문건수(NO 최댓값) + 검증용.
- fee_breakdown = C-1~C-14 매핑.

**땡겨요** (신규 `ddangyo_service`) — .xls (xlrd):
- `요약`: (A)주문결제(=매출) (D)차감금액(=총비용) (E)정산금액(=정산).
- 파일명이 `건별`/`일별` 두 형태 → 상세 시트 파싱은 건수 산출용, 요약이 SSOT.
- fee_breakdown = 차감 항목(있는 만큼).

### 2.3 라우터 / UI
- 백엔드: 채널별 `POST /{channel}/sync/monthly-excel/upload` (기존 쿠팡 패턴 복제).
  요기요·땡겨요는 신규 라우터, 배민은 기존 라우터에 복호화 반영.
- 프론트: `RevenueManagement` 각 채널 카드에 "엑셀 업로드+적재 + 비번란"(쿠팡/배민 UI 재사용).

### 2.4 검증
- 파서 단위 테스트: 실제 샘플(1~3월) fixture 로 매출/총비용/정산/건수 기대값 검증.
  - 요기요 1월: 매출 424,500 / 차감 215,458 / 정산 209,042.
  - 배민 5월: 상세 합 기반 매출·정산·총비용 (요약 H=1,847,541 입금).
  - ⚠️ 샘플 xlsx는 민감자료(`2026소득분석/`, git 제외) → fixture 는 **익명화 축약본**을 저장소에 커밋.

## 파트 3 — 엑셀 정산액 ↔ 은행 실입금 대조

### 3.1 데이터
- **엑셀 정산액**: `DeliveryRevenue(source in excel).settlement_amount` (채널·월).
- **은행 실입금**: `BankTransaction` 입금건을 `_resolve_settlement`(내용/적요 → 채널)로
  채널 매칭 후 (채널, 입금월) 합계. bank_sync 로직 재사용.
  - 신한 엑셀 컬럼: 거래일자·적요·입금(원)·내용. 내용 상호명으로 매칭
    (쿠팡페이→쿠팡, 우아한형제들/배민→배민, 위대한상상/요기요→요기요, 땡겨요→땡겨요).

### 3.2 판정 (±1%)
```
delta = 실입금 − 엑셀정산
비율  = |delta| / 엑셀정산
일치(✅): 비율 ≤ 1%
불일치(⚠️): 그 외
```

### 3.3 불일치 이유 자동 분석
- 실입금 0 / 매칭 0건 → "통장에서 해당 채널 입금 못 찾음 (적요 확인 필요)"
- 실입금 < 엑셀(−1% 초과) → "정산 일부 미입금 또는 익월 입금 가능"
- 실입금 > 엑셀(+1% 초과) → "전월 이월 입금 포함 가능"
- 항상 `delta`·`비율`·매칭 입금건수를 함께 제공.

### 3.4 API / UI
- 백엔드: `GET /revenue/settlement-reconciliation?year=` →
  `[{channel, year, month, excel_settlement, bank_deposit, delta, rate, status, reason, matched_count}]`.
- 프론트: 배달앱 매출표 **정산 셀에 배지(✅/⚠️)** + 클릭 시 상세(엑셀정산/실입금/차이/이유).

### 3.5 검증
- 단위 테스트: 일치/미달/초과/미매칭 4케이스의 status·reason.

- `DeliveryRevenue` 스키마 변경 없음 (기존 필드 재사용).
- 채널 자격증명(배민 등)에 `excel_password_encrypted` 컬럼 추가 가능성(자동수집 단계) —
  이번엔 업로드 시 비번 입력이 우선, 저장은 후속 옵션.

## 빌드 순서 (파트별 독립 검증)
1. **파트 1**: 정리 스크립트(전/후 비교→승인→실행) + 결정적 병합 + 테스트.
2. **파트 2**: 배민 복호화 재구현 → 요기요 → 땡겨요 파서 + 업로드 UI + fixture 테스트.
3. **파트 3**: 대조 API + 배지 UI + 테스트.

## 리스크 / 유의
- 민감 샘플(급여·매출 raw)은 git 제외. 테스트 fixture 는 익명화 축약본만 커밋.
- 배민 비번 등 비밀값은 문서·로그·응답 미노출, 암호화 저장.
- 정리 스크립트는 되돌릴 수 있게 삭제 전 백업(카운트+금액 로그, dry-run 우선).
- ±1% 임계는 상수로 두어 조정 가능.
